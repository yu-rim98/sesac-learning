## 프로젝트 개요

### 기능

- 사용자 관리 : 회원가입 및 로그인 기능 (Spring Security 활용)
- 게시글 : 이미지와 텍스트를 포함한 게시글 생성 및 조회 기능
- 소셜 :
  - 사용자 간 팔로우/언팔로우 기능
  - 게시글에 대한 좋아요 기능
  - 게시글에 대한 댓글 작성 기능
- 프로필 : 사용자 프로필 정보(이름, 소개, 이미지) 조회 및 수정 기능
- 탐색 및 검색 : 사용자 이름 또는 게시글 기반의 검색 기능

> JPA를 통해 데이터를 관리하고, 타임리프를 사용해 서버 사이드 렌더링 방식으로 웹 UI를 구성하는 기본적인 MVC 구조의 프로젝트

## 프로젝트 좋았던 점

#### 1. 서비스 계층의 인터페이스화

- `PostService`, `UserService` 등 서비스 계층에 인터페이스(추상체)와 구현체 분리
- SOLID의 의존성 역전 원칙(DIP)을 적용해 향후 다른 구현체로 쉽게 교체할 수 있는 유연한 구조를 갖는다.
  - DIP는 변하기 쉬운 구체적인 구현체가 아니라 변하지 않는 추상화에 의존해야 한다는 원칙이다.
  - 이를 통해 상위 모듈과 하위 모듈 간의 결합도를 낮추고 **의존성의 방향을 역전**시켜 확장에 열려있는 유연한 아키텍처를 만들 수 있다.

#### 2. DTO(Data Transfer Object) 활용

- 컨트롤러 단에서 `SignUpRequest`, `PostResponse`와 같은 DTO를 사용해 계층 간 데이터 전달 역할을 명확하게 분리
- Entity를 직접 노출하지 않고 필요한 데이터만 가공해 전달하는 것은 보안과 시스템 유연성 측면에서 중요하다.

#### 3. 전역 예외 처리

- GlobalExceptionHandler를 만들어서 애플리케이션 전역에서 발생하는 예외를 한 곳에서 처리하도록 설계
- 이는 코드 중복을 줄이고 오류 처리 정책을 일관성 있게 가져갈 수 있다.

## 프로젝트 개선 방향

#### 1. 서비스의 과도한 책임

- 서비스 클래스가 너무 많은 책임을 지고 있다는 생각이 든다.
  - 예를 들어 프로필 수정하는 로직이 '사용자 정보 변경'이랑 '파일 처리'의 두가지 책임을 가지고 있기 때문에 단일 책임 원칙(SRP)을 위반

#### 2. 빈약한 도메인 모델

- Entity 객체가 스스로 자신의 상태를 관리하고 비즈니스 로직을 수행하는 것이 아니라 서비스 계층에서 해당 객체의 데이터를 꺼내 모든 것을 처리하고 있다. 이는 절차지향적 코드와 다를 바가 없다.
